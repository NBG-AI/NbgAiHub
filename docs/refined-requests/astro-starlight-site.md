# Refined Request: Astro Starlight Site for NbgAiHub

## Category

Development (frontend / static site). Sibling workspace to existing `pipeline/`.

## Objective

Build the Astro Starlight web UI at `site/` that renders the hub's five pillars (Skills, Tips, News, Journeys, Glossary) plus supporting pages, reads content via Astro 5's `glob()` loader directly from the existing repo content folders, exposes a left sidebar, ships built-in search (Pagefind), and runs locally via `npm run dev` on port 4321. Hosting is explicitly deferred — the MVP success criterion is "working dev server with all sidebar entries clickable and the news collection rendered from `/news/published/`."

## Scope

### In scope

- `site/` workspace at repo root (sibling to `pipeline/`) — own `package.json`, `tsconfig.json`, `node_modules/`, `astro.config.mjs`, `content.config.ts`.
- Astro 5.x + `@astrojs/starlight` (latest at install time).
- TypeScript strict, ESM, Node 22 LTS (matching `pipeline/.nvmrc`).
- 5 content collections via Astro 5 `glob()` loader pointing at existing repo folders:
  - `news` ← `../news/published/*.md` (may be empty until PR #1 merges)
  - `skills` ← `../skills/*.md` (folder will be created empty)
  - `tips` ← `../tips/*.md` (folder will be created empty)
  - `glossary` ← `../glossary/*.md` (folder will be created with ~5 seed terms)
  - `journeys` ← `../journeys/*.md` (folder will be created with a `day-1.md` placeholder)
- Zod schemas matching DECISIONS.md "Shared content shape" (12 keys + news-specific `source` + `fingerprint`).
- Sidebar nav with 9 entries: Home, Start Here (Day 1, Week 1), News, Skills, Tips & Tricks, Glossary, Reference, Contribute.
- 8 pages: `/`, `/news`, `/news/<slug>` (dynamic), `/skills`, `/tips`, `/glossary`, `/reference`, `/contribute`, `/start-here/day-1`.
- 6 custom components: `HomeHero`, `NewsPanel`, `NewsList`, `AudienceBadge`, `SkillCard`, `AudienceFilter`.
- Empty-state fallbacks for each collection page when the source folder is empty.
- Beginner/Advanced/Both client-side filter (pure DOM toggle via CSS class — no fetch, no framework).
- Pagefind search active (Starlight default).
- Custom CSS for hero, news cards, audience badge colors. Otherwise Starlight default theme.

### Out of scope

- Deployment to any host (GH Pages, Vercel, Netlify, Cloudflare). Hosting remains an open question in SCOPE.md.
- The hub-as-skill Claude plugin (separate workstream).
- Real Day 1 journey content (placeholder only).
- Hero image extraction for news items (deferred per SCOPE.md; site renders `hero_image` if present in frontmatter, gracefully ignores if absent — forward compat).
- Any modification to `pipeline/`, `news/published/`, `config/`, or any non-`site/` content.
- Authentication, comments, analytics, multi-language, mobile-app rendering tweaks beyond Starlight's responsive defaults.

## Requirements

### Functional

1. **F1 — Workspace scaffolding.** `site/` workspace contains `package.json` declaring `astro` and `@astrojs/starlight` as direct deps, a `tsconfig.json` extending Starlight's recommended strict TS config, `astro.config.mjs`, `content.config.ts`, `src/`, and `public/`.

2. **F2 — Content collections.** `content.config.ts` defines 5 collections (`news`, `skills`, `tips`, `glossary`, `journeys`), each using Astro 5's `glob()` loader against the corresponding `../<folder>/*.md` path with a Zod schema matching the canonical shape. News schema adds `source` and `fingerprint`. Optional `hero_image: string().url().optional()` permitted on news for forward compatibility.

3. **F3 — Sidebar.** `astro.config.mjs` configures the Starlight sidebar with 9 entries in this order: Home, Start Here (collapsible group containing Day 1 and Week 1), News, Skills, Tips & Tricks, Glossary, Reference, Contribute.

4. **F4 — Homepage (`/`).** Renders `HomeHero` (title, tagline, two primary CTAs: "Start Here → Day 1" and "Browse Skills") followed by `NewsPanel` (5 most recent published news items, compact cards) and a row of "featured" cards for Tips, Skills, Glossary.

5. **F5 — `/news` index.** Renders `NewsList` showing all items in the `news` collection, sorted by `authored` descending, with `AudienceFilter` and topic filters at the top.

6. **F6 — `/news/<slug>` per-item pages.** Auto-generated by Astro from the `news` collection. Each page shows title, `AudienceBadge`, topic chips, source name, "Read on source ↗" link to `external_link`, and the AI summary as body.

7. **F7 — Catalog pages.** `/skills`, `/tips` render card grids of their respective collections (with empty-state fallback). `/glossary` renders a single page with anchor links per term (per DECISIONS.md "Hybrid glossary"). `/reference` is a hand-authored markdown cheatsheet. `/contribute` is a hand-authored markdown page describing PR contribution flow.

8. **F8 — `/start-here/day-1` placeholder.** Renders a page with headings for each of the 6 designed Day 1 steps plus "coming soon" body content. Content is captured in SCOPE.md MVP table; this phase only builds the page shell.

9. **F9 — Empty-state fallbacks.** When `news/published/`, `skills/`, `tips/`, `glossary/`, or `journeys/` is empty, the corresponding page renders a friendly "No items yet — see Contribute for how to add one" message (no crash, no empty grid). The homepage `NewsPanel` shows a similar fallback when news is empty.

10. **F10 — Beginner/Advanced filter.** `AudienceFilter` renders three checkboxes (Beginner, Advanced, Both — all checked by default). Toggling a checkbox hides/shows cards by CSS class on their `data-audience` attribute. Client-side only. Persists state via `localStorage`.

11. **F11 — Search.** Pagefind index built at `npm run build` time; search bar in Starlight header is functional.

12. **F12 — Strict frontmatter validation.** A markdown file in any source folder that fails its collection's Zod schema must cause `astro check` (or `npm run build`) to fail with a clear error naming the invalid file and field. No silent skipping.

### Non-functional

13. **NF1 — Astro 5.x + Starlight latest 0.x at install time.**
14. **NF2 — TS strict** (`"strict": true`, `"noUncheckedIndexedAccess": true`).
15. **NF3 — ESM only.** `"type": "module"` in `package.json`.
16. **NF4 — Node 22.** `site/.nvmrc` set to `22`.
17. **NF5 — `astro check` exits 0.**
18. **NF6 — `npm run build` exits 0** producing `dist/` with static HTML + Pagefind index.
19. **NF7 — Dev port 4321.** `astro.config.mjs` pins dev server port to 4321 per CLAUDE.md → Ports.
20. **NF8 — No fallback config.** Schema validation failures, missing dependencies, and missing config files must fail loudly. No defaults that mask real misconfiguration.

## Constraints

- All commands run from `site/` working directory.
- Cross-workspace coupling minimized — the frontmatter shape is shared with the pipeline but the schema is duplicated in `content.config.ts` (no shared TypeScript module). Future refactor possible if drift becomes a problem.
- Site is read-only against repo content folders — it never writes to `news/`, `skills/`, etc.
- No deployment YAML, no hosting setup.
- Singular DB table names rule (N/A — no DB).
- No version-control operations from the site code itself.
- Hub tone: *"what I wish I knew a year ago"* — opinionated, plainspoken (carry into all hand-authored content like Reference and Contribute pages).

## Assumptions

- **A1 — Astro version: `^6.x`** at install time. **SUPERSEDES original "Astro 5.x" assumption** — Astro 6 went stable 2026-03-10 and Starlight 0.39 requires it. Zero migration cost since we're greenfield. See DECISIONS.md "Astro 5 → 6" supersession.
- **A2 — Starlight version: `^0.39.x`** at install time. **SUPERSEDES original "latest 0.x" assumption** — version pinned because 0.39 is the version that requires/supports Astro 6. Earlier 0.38 dropped Astro 5 support; 0.39 jumped to Astro 6.
- **A3 — Sidebar configured declaratively** in `astro.config.mjs` via Starlight's `sidebar:` option (not auto-generation, since we want explicit control over labels, order, and the Start Here grouping).
- **A4 — Frontmatter schema duplicated, not imported.** The Zod schema for the news collection lives in `site/content.config.ts` and is a logical mirror of the pipeline's schema in `pipeline/src/frontmatter.ts`. No cross-workspace TypeScript import. Drift risk is acknowledged and accepted for MVP; a future "shared schema package" can be extracted if drift becomes painful. Rationale: cross-workspace TS imports require either a monorepo tool (npm workspaces, pnpm, turbo) or duplicated type declarations — added complexity not justified for MVP.
- **A5 — Slug for `/news/<slug>` routes** = the part of the filename between the date prefix and the `.md` extension. Example: `2026-05-18-claude-code-helped-me.md` → `/news/claude-code-helped-me`. Date is rendered in the page body, not the URL.
- **A6 — Custom CSS is minimal:** only home hero layout, news card styling, and audience badge colors. ~100 lines max in `site/src/styles/custom.css`. Everything else uses Starlight defaults.
- **A7 — Audience badge colors:** Beginner = green (`#0a7`), Advanced = orange (`#e60`), Both = blue (`#08c`). Tweakable later.
- **A8 — Empty-state copy** is consistent across collections: *"No items yet. See [Contribute](/contribute) for how to add one."*
- **A9 — No unit tests for components in MVP.** Verification is via `astro check`, `npm run build`, and manual click-through. Adding component tests (e.g., via `@playwright/test`) is deferred. Rationale: the pipeline's tests cover frontmatter shape; the site mostly transforms that shape into HTML, and Astro's build itself catches schema errors. Component-level testing in Astro is also relatively immature in 2026.
- **A10 — No third-party integrations:** no analytics (Plausible/GA), no image CDN, no comments (Giscus), no error tracking (Sentry). Deferred.
- **A11 — Sidebar order:** Home → Start Here (Day 1, Week 1) → News → Skills → Tips & Tricks → Glossary → Reference → Contribute. Order matters; reflects newcomer journey then catalog then meta.
- **A12 — Audience filter uses checkboxes** in a small toolbar at the top of `/news` and the homepage News panel. Pure client-side, no SSR concerns. State persisted to `localStorage` so the user's preference survives page reloads.
- **A13 — Pagefind is the search backend.** Starlight ships with Pagefind enabled by default at build time; no extra config needed.
- **A14 — Strict-mode build behavior.** A markdown file failing schema validation breaks `astro check` and `npm run build` with a named-file error. No silent skipping.
- **A15 — Seed content for empty collections:**
  - `skills/.gitkeep` only (catalog empty for now)
  - `tips/.gitkeep` only
  - `glossary/` seeded with ~5 minimal markdown files: `claudemd.md`, `mcp.md`, `skill.md`, `plugin.md`, `agent.md` (each: term, 1-paragraph definition, deeper_link placeholder)
  - `journeys/day-1.md` with the 6 step headings and "coming soon" body content
- **A16 — `hero_image` is optional, forward-compatible.** Adding it to the schema today means future hero-image extraction by the pipeline plugs in with no site change.
- **A17 — Local-dev-only.** No CI deploy YAML. No hosting integration. `dist/` is git-ignored. The user runs `npm run dev` to view the site; running `npm run build && npm run preview` validates the production output without deploying anywhere.
- **A18 — Test runner reuse.** If a test framework is needed for any pure utility (slug helpers, schema helpers), reuse Vitest (already in `pipeline/`). Otherwise omit.

## Open questions

- **OQ1 — Hosting** (carried from SCOPE.md): GH Pro for Pages vs Vercel/Netlify/Cloudflare free vs defer. **Resolution: defer until MVP demo is complete.**
- **OQ2 — Custom branding.** Use Starlight default theme; no logo, no custom font, no color palette beyond the audience badges. Revisit before any public-facing demo. **Resolution: defer.**
- **OQ3 — Skill catalog data model.** The skills frontmatter schema is duplicated from the canonical shape; real skill entries may surface additional fields (e.g., `install_command`, `repo_url`). **Resolution: keep schema minimal for now; extend when the first real skill is added.**
